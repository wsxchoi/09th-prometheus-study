# Prometheus mmap Head Chunks 코드로 이해하기

> Memory Footprint, Memory-Mapped I/O, Offset — WAL의 쓰기 경로에서 Sequential I/O, Page, Write Amplification, Atomic Operation을 살펴봤다면, 이번에는 **"다 쓴 청크가 메모리에서 내려가고, 다시 읽히기까지"** 의 경로를 따라가면서 OS 핵심 개념 3가지가 실제 프로덕션 코드에서 어떻게 쓰이는지를 살펴본다.

---

## 전체 흐름: 데이터 라이프사이클

Prometheus는 모니터링 대상을 주기적으로 스크레이핑해서 샘플(시간 + 값)을 수집한다. 이 샘플이 디스크에 영구 저장되기까지:

![image.png](https://github.com/user-attachments/assets/a2fcf176-69b5-4c56-9694-5b2d1b80a2c9)

```
샘플 수집 → [1] WAL 기록 → [2] Head 블록 인메모리 저장
         → [3] immutable 청크를 mmap 파일로 flush → [4] 블록 압축
```

이 글에서는 **[3] immutable 청크를 mmap 파일로 flush** 단계를 다룬다.

### 청크와 파일의 관계

**청크마다 파일을 하나씩 만드는 게 아니라, 여러 청크를 하나의 파일에 이어 붙여 저장한다.** 시리즈가 10만 개라면 청크도 수십만 개인데, 파일을 청크마다 하나씩 만들면 OS의 파일 디스크립터가 바닥나고 공간 낭비도 심하다. 그래서 `chunks_head/` 디렉토리에 최대 128MB짜리 파일을 만들고, 그 안에 청크를 기차 칸처럼 이어 붙인다.

```
data/
├── chunks_head/           ← mmap Head Chunks 영역
│   ├── 000001             ← chunk 파일 ([헤더][ChunkA][ChunkB][ChunkC]...)
│   └── 000002             ← 000001이 128MB 차면 새 파일
└── wal/                   ← WAL 영역
    ├── checkpoint.000003
    ├── 000004             ← segment (코드상 명칭)
    └── 000005
```

WAL의 파일(`segment` 타입, `tsdb/wal/wal.go`)과 달리 `chunks_head/`의 파일들은 코드상 그냥 **"file"** 이다(`tsdb/chunks/head_chunks.go`).

## 1. Memory Footprint — 왜 디스크로 내려야 하는가

**Memory Footprint**란 프로세스가 실제로 사용하고 있는 물리 메모리의 총량이다. 흔히 **RSS(Resident Set Size)** 로 측정한다.

> **RSS**: 프로세스가 현재 물리 RAM에 실제로 올려놓고 있는 메모리의 크기. `malloc`으로 할당해도 실제 접근 전까지는 물리 메모리에 배정되지 않는다. Linux에서는 `top`의 RES 컬럼이나 `/proc/<pid>/status`의 `VmRSS`로 확인한다.

**VSS(Virtual Set Size)** 와 **RSS** 는 다르다. mmap으로 128MB 파일을 매핑하면 VSS는 128MB 늘어나지만, 실제로 접근한 페이지만 RSS에 반영된다.

### 청크가 메모리를 얼마나 먹는가

XOR 압축된 청크 하나는 약 120~200바이트의 `data`를 담고 있다. Go 런타임 슬라이스 헤더(24바이트)와 메타데이터를 더하면 청크당 약 **150~250바이트**를 Heap에서 차지한다.

시리즈 10만 개, retention 2시간, 스크레이핑 간격 15초 기준:

```
청크당 샘플 수: ~120개
2시간 동안 샘플 수: 7,200 / 15 = 480개
시리즈당 청크 수: 480 / 120 = 4개
전체 청크 수: 100,000 × 4 = 400,000개
청크당 메모리: ~200바이트

총 메모리: 400,000 × 200 = ~80MB (청크 데이터만)
```

이 400,000개 청크 중 **현재 쓰기 중인 것은 시리즈당 1개, 즉 100,000개뿐**이다. 나머지 300,000개는 이미 full이 되어 immutable한 상태 — 읽기만 가능한 데이터를 Heap에 들고 있는 것은 낭비다.

### mmap 청크로 교체하면

immutable 청크를 디스크로 내리고, 메모리에는 **참조 정보만** 남긴다. 이 참조 정보는 mmap region이 아니라 **Heap**에 저장된다(인메모리 인덱스의 시리즈별 위치 목록).

![image.png](https://github.com/user-attachments/assets/c686d71c-1836-4f27-a47a-e497cc587df2)

```
참조 정보: chunk ref(8B) + mint(8B) + maxt(8B) = 24바이트

기존: 300,000 × 200B = 60MB
변경: 300,000 × 24B  = ~7.2MB

절감: ~53MB (청크 데이터 기준 ~88% 감소)
```

Head 블록 전체 RSS 기준으로는 인덱스와 심볼 등 다른 부분도 있으므로, 실제 환경에서는 **15~50% 수준의 메모리 절감**이 관찰된다.

## 2. Memory-Mapped I/O — 어떻게 내리면서도 빠르게 읽는가

mmap이 빛나는 조건:
1. **읽기 위주**: 쓰기는 flush 시 한 번, 이후로는 읽기만
2. **랜덤 접근**: 쿼리에 따라 아무 청크나 접근할 수 있어야 함
3. **희소 접근**: 전체 파일 중 일부 청크만 읽는 경우가 대부분

mmap Head Chunks는 이 세 가지를 모두 충족한다. 청크를 flush한 뒤에는 immutable이니 읽기만 하고, 쿼리가 요청하는 시리즈의 청크만 랜덤으로 접근하며, 대부분의 쿼리는 전체 청크 중 극히 일부만 건드린다.

## 3. Offset — mmap된 파일에서 특정 청크를 어떻게 찾는가

**Offset(오프셋)** 이란 기준점으로부터의 거리다. 파일에서의 오프셋은 "파일 시작점에서 몇 바이트 떨어져 있는가"를 뜻한다.

### mmap에서 오프셋이 핵심인 이유

```
base = mmap(fd, fileSize)

파일의 offset 1234에 있는 데이터
= 메모리의 base + 1234에 있는 데이터
= base[1234]
```

파일 오프셋이 곧 메모리 인덱스가 된다. `lseek()` + `read()`로 파일 위치를 옮기고 읽을 필요 없이, 배열 인덱싱 한 번으로 끝난다.

### 청크 참조(Reference)의 구조

특정 청크에 접근하려면 두 가지 정보만 필요하다:
1. **어떤 파일에 있는가** → 파일 번호 (예: `000093`)
2. **파일의 어디에 있는가** → 파일 내 오프셋 (예: 바이트 `1234`)

### 청크 디스크 포맷

```
파일 구조:
[8B 헤더] [청크1] [청크2] ... [청크N]

각 청크:
┌─────────────────────┬──────┬──────┬──────────┬─────┬──────┬───────┐
│ series ref (8B)     │ mint │ maxt │ encoding │ len │ data │ CRC32 │
│                     │ (8B) │ (8B) │ (1B)     │(var)│(var) │ (4B)  │
└─────────────────────┴──────┴──────┴──────────┴─────┴──────┴───────┘
```

오프셋이 `series ref`의 첫 바이트를 가리키면, 고정 길이 필드를 순서대로 읽고, `len`으로 `data` 크기를 알아내고, CRC32를 검증한다. **별도의 인덱스 없이 오프셋 하나로 청크 전체를 파싱할 수 있는 자기 서술적(self-describing) 포맷**이다.

### Immutability의 역할

오프셋 기반 접근이 안전하려면 **청크가 쓰인 후 위치가 바뀌면 안 된다.** mmap Head Chunks는 append-only로 쓰이고, 한번 쓰인 청크는 삭제될 때까지 그 자리에 있다. "immutability"는 단순히 메모리 절약을 위한 것이 아니라, **오프셋 기반 접근의 정합성을 보장하는 전제 조건**이기도 하다.

## 크래시 복구: 인메모리 인덱스 재구성

Heap에 들고 있는 주소록(인메모리 인덱스)은 프로세스가 죽으면 날아간다. 복구에 필요한 모든 정보는 **디스크에 이미 있다.**

```
[1단계] chunks_head/ 파일 스캔
    │  청크의 실제 data는 읽지 않고 series ref, mint, maxt만 읽어서
    │  "시리즈 A → [파일5/오프셋100, 파일5/오프셋400]" 맵을 구축
    │  → 주소록(인메모리 인덱스) 복원
    ▼
[2단계] WAL 리플레이
    │  아직 full이 안 된 청크의 샘플들을 복원
    │  1단계 맵을 활용해 mmap 청크 범위의 샘플은 건너뜀
    ▼
[복구 완료]
    Heap: 인덱스 + 활성 청크
    mmap: immutable 청크 파일
```

1단계에서 메타데이터만 읽으므로 수십만 개 청크를 훑어도 빠르게 끝나고, 2단계에서 mmap 청크 덕분에 WAL 리플레이 대부분을 건너뛸 수 있어 **시작 시간이 15~30% 단축**된다.
