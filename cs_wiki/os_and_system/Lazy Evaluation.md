## 1. Lazy Evaluation / Lazy Loading (지연 평가/로딩)

---

### OS 관점

OS에서 "lazy"는 **리소스 할당을 최대한 미뤄서 효율성을 극대화**하는 핵심 설계 철학입니다.

### 1.1 Demand Paging (요구 페이징)

가장 대표적인 lazy loading 구현입니다.

```
프로세스 실행 시작
     │
     ▼
┌─────────────────────────────────────────────────┐
│ 전체 프로그램을 메모리에 로드?  → NO!            │
│ 실제 접근하는 페이지만 로드    → YES!           │
└─────────────────────────────────────────────────┘

[Virtual Memory]          [Physical Memory]        [Disk]
┌──────────┐              ┌──────────┐           ┌──────────┐
│ Page 0   │──valid───────│ Frame 2  │           │ Page 0   │
│ Page 1   │──invalid─────│          │◄──────────│ Page 1   │
│ Page 2   │──invalid─────│ Frame 5  │           │ Page 2   │
│ Page 3   │──valid───────│          │           │ Page 3   │
└──────────┘              └──────────┘           └──────────┘
    │                          ▲
    │ Page Fault 발생 시       │
    └──────────────────────────┘
         이때서야 로드 (lazy!)
```

**Page Fault 처리 흐름:**

1. CPU가 invalid 페이지 접근
2. MMU가 page fault 트랩 발생
3. OS가 디스크에서 해당 페이지 로드
4. 페이지 테이블 업데이트 (invalid → valid)
5. 중단된 명령어 재실행

```c
// 리눅스 커널의 page fault handler (단순화)
static vm_fault_t handle_pte_fault(struct vm_fault *vmf) {
    if (!pte_present(vmf->orig_pte)) {
        // 페이지가 메모리에 없음 → lazy loading 수행
        return do_fault(vmf);  // 디스크에서 로드
    }
    // ...
}
```

### 1.2 Copy-on-Write (COW)

`fork()` 시 메모리 복사를 lazy하게 처리합니다.

```
fork() 직후:
┌─────────────┐     ┌─────────────┐
│   Parent    │     │    Child    │
│ Page Table  │     │ Page Table  │
└──────┬──────┘     └──────┬──────┘
       │                   │
       │  (둘 다 READ-ONLY로 마킹)
       ▼                   ▼
    ┌─────────────────────────┐
    │   Shared Physical Page  │  ← 실제 복사 안 함!
    └─────────────────────────┘

Child가 write 시도할 때:
       │
       ▼
    Page Fault (Protection Fault)
       │
       ▼
    ┌─────────────────────────────┐
    │ OS: "아, 이제 진짜 복사해야겠네" │
    │     → 새 페이지 할당 & 복사    │
    └─────────────────────────────┘
```

**장점:**

- `fork()` + `exec()` 패턴에서 대부분의 페이지는 복사 불필요
- 메모리와 시간 모두 절약

### 1.3 Lazy Allocation (지연 할당)

`malloc()`이나 `mmap()`은 즉시 물리 메모리를 할당하지 않습니다.

```c
// 1GB 할당 요청
char *buf = malloc(1024 * 1024 * 1024);  // 거의 즉시 반환!

// 실제로는 가상 주소 공간만 예약됨
// 물리 메모리는 0바이트 사용 중

buf[0] = 'A';        // 첫 페이지만 할당됨 (4KB)
buf[500000000] = 'B'; // 해당 페이지만 할당됨 (4KB)

// 총 물리 메모리 사용: 8KB (1GB 아님!)
```

```
Virtual Address Space        Physical Memory
┌────────────────────┐      ┌─────────────┐
│ 0x0000: Reserved   │      │             │
│ ...                │      │ Frame 100 ◄─┼── 실제 접근한
│ 0x7fff: Allocated  │──────│ Frame 101   │   페이지만 할당
│        (1GB 영역)   │      │             │
│ ...                │      │   (대부분    │
│                    │      │    비어있음) │
└────────────────────┘      └─────────────┘
```

### 1.4 Lazy TLB Invalidation

멀티코어 환경에서 TLB shootdown을 최적화합니다.

```
Core 0: 페이지 테이블 수정
        │
        ├── 즉시 모든 코어에 IPI 전송? → 비용 큼
        │
        └── Lazy 방식:
            - 해당 페이지를 "needs flush" 마킹
            - 다른 코어가 그 페이지 접근 시 flush
            - 접근 안 하면 flush 안 함
```

### DB 관점

### 1.5 Lazy Loading in ORM

관계형 데이터를 필요할 때만 로드합니다.

```python
# SQLAlchemy 예시
class User(Base):
    __tablename__ = 'users'
    id = Column(Integer, primary_key=True)
    posts = relationship("Post", lazy="select")  # lazy loading

# 쿼리 실행
user = session.query(User).get(1)
# SQL: SELECT * FROM users WHERE id = 1

# posts에 접근할 때 비로소 추가 쿼리
for post in user.posts:
# SQL: SELECT * FROM posts WHERE user_id = 1  ← 이 시점에 실행
    print(post.title)
```

**Lazy Loading 전략들:**

| 전략 | 동작 | 사용 시점 |
| --- | --- | --- |
| `select` (lazy) | 접근 시 SELECT 쿼리 | 기본값, 관계 데이터 자주 안 씀 |
| `joined` (eager) | JOIN으로 한번에 로드 | 항상 함께 쓰는 데이터 |
| `subquery` | 서브쿼리로 로드 | 컬렉션 로드 시 N+1 방지 |
| `selectin` | IN 절로 배치 로드 | 여러 객체의 관계 로드 |

**N+1 문제 (Lazy Loading의 함정):**

```python
users = session.query(User).all()  # 1번 쿼리
for user in users:
    print(user.posts)  # N번 쿼리 (user마다 1번씩)
# 총 N+1번 쿼리 → 성능 저하
```

### 1.6 Lazy Materialization (지연 실체화)

컬럼 스토어 DB에서 쿼리 결과를 lazy하게 구성합니다.

```
Traditional (Early Materialization):
┌─────────────────────────────────────────────┐
│ SELECT name, age FROM users WHERE age > 20  │
└─────────────────────────────────────────────┘
        │
        ▼
┌─────────────────────┐
│ 1. age 컬럼 스캔     │
│ 2. name 컬럼 스캔    │
│ 3. 즉시 튜플 조합    │ ← 모든 행을 미리 조합
│ 4. 필터링           │
└─────────────────────┘

Lazy Materialization:
┌─────────────────────┐
│ 1. age 컬럼만 스캔   │
│ 2. 조건 만족하는     │
│    position 집합 획득│
│ 3. 해당 position의   │ ← 필요한 것만 나중에 조합
│    name만 가져옴     │
└─────────────────────┘
```

### 1.7 Deferred Constraints (지연 제약조건)

트랜잭션 커밋 시점까지 제약조건 검사를 미룹니다.

```sql
-- 순환 참조 문제
-- Employee A의 manager는 B, B의 manager는 A

SET CONSTRAINTS ALL DEFERRED;

INSERT INTO employees (id, name, manager_id) VALUES (1, 'Alice', 2);
-- 이 시점에 manager_id=2가 없어도 OK (검사 지연)

INSERT INTO employees (id, name, manager_id) VALUES (2, 'Bob', 1);
-- 이 시점에도 OK

COMMIT;  -- 이때 모든 제약조건 검사 → 통과!
```

## 2. Memory-mapped Index (메모리 맵 인덱스)

---

### OS 관점

### 2.1 mmap 시스템 콜 심층 분석

```c
void *mmap(
    void *addr,      // 원하는 시작 주소 (보통 NULL)
    size_t length,   // 매핑 크기
    int prot,        // 보호 모드 (PROT_READ, PROT_WRITE, PROT_EXEC)
    int flags,       // 매핑 타입 (MAP_SHARED, MAP_PRIVATE)
    int fd,          // 파일 디스크립터
    off_t offset     // 파일 내 오프셋
);
```

**매핑 타입 비교:**

```
MAP_SHARED:
┌─────────────┐     ┌─────────────┐
│  Process A  │     │  Process B  │
│  VA: 0x1000 │     │  VA: 0x2000 │
└──────┬──────┘     └──────┬──────┘
       │                   │
       └────────┬──────────┘
                ▼
        ┌─────────────┐
        │Physical Page│ ──── 변경사항이 파일에 반영됨
        └─────────────┘
                │
                ▼
        ┌─────────────┐
        │  Disk File  │
        └─────────────┘

MAP_PRIVATE (Copy-on-Write):
┌─────────────┐
│  Process A  │
│  VA: 0x1000 │
└──────┬──────┘
       │ write 시
       ▼
┌─────────────┐
│ Private Copy│ ──── 변경사항이 파일에 반영 안 됨
└─────────────┘
```

### 2.2 Page Table과 mmap의 관계

```
mmap() 호출 시:
┌────────────────────────────────────────────────────────┐
│ 1. VMA (Virtual Memory Area) 생성                       │
│    - vm_start, vm_end 설정                              │
│    - vm_file = 매핑된 파일                               │
│    - vm_ops = 파일시스템별 핸들러                         │
│                                                        │
│ 2. Page Table Entry는 아직 invalid!                     │
│    (실제 물리 메모리 할당 없음 - lazy!)                    │
└────────────────────────────────────────────────────────┘

메모리 접근 시:
┌────────────────────────────────────────────────────────┐
│ 1. Page Fault 발생                                     │
│ 2. fault handler가 VMA 찾음                            │
│ 3. Page Cache에서 해당 페이지 찾기                       │
│    - 있으면: 해당 페이지 매핑                            │
│    - 없으면: 디스크에서 읽어서 Page Cache에 추가          │
│ 4. Page Table Entry 업데이트 (valid로)                  │
└────────────────────────────────────────────────────────┘
```

### 2.3 Page Cache와의 상호작용

```
                    ┌──────────────────┐
                    │    User Space    │
                    │  (mmap region)   │
                    └────────┬─────────┘
                             │
                    ┌────────▼─────────┐
                    │   Page Table     │
                    └────────┬─────────┘
                             │
                    ┌────────▼─────────┐
                    │   Page Cache     │ ← 핵심!
                    │  (커널 메모리)    │
                    └────────┬─────────┘
                             │
                    ┌────────▼─────────┐
                    │   Block Layer    │
                    └────────┬─────────┘
                             │
                    ┌────────▼─────────┐
                    │      Disk        │
                    └──────────────────┘
```

**중요 포인트:**

- mmap은 Page Cache를 직접 유저 공간에 매핑
- `read()` 시스템 콜은 Page Cache → User Buffer 복사 필요
- mmap은 이 복사를 제거 (zero-copy)

### 2.4 mmap vs read/write 성능 비교

```
read() 방식:
┌──────────┐    ┌────────────┐    ┌─────────────┐
│   Disk   │───▶│ Page Cache │───▶│ User Buffer │
└──────────┘    └────────────┘    └─────────────┘
                     복사 발생!

mmap() 방식:
┌──────────┐    ┌────────────────────────────┐
│   Disk   │───▶│ Page Cache = User에서 직접  │
└──────────┘    │            접근 가능        │
                └────────────────────────────┘
                     복사 없음!
```

**성능 특성:**

| 상황 | read/write | mmap |
| --- | --- | --- |
| 순차 읽기 (대용량) | 더 좋음 (read-ahead 최적화) | 페이지 폴트 오버헤드 |
| 랜덤 읽기 | 시스템 콜 오버헤드 | 더 좋음 |
| 작은 파일 빈번 접근 | 시스템 콜 비용 | 더 좋음 |
| 여러 프로세스 공유 | 각자 버퍼 필요 | 메모리 공유 가능 |

### DB 관점

### 2.5 Buffer Pool vs mmap 논쟁

DB에서 가장 뜨거운 설계 논쟁 중 하나입니다.

**Buffer Pool 방식 (전통적):**

```
┌─────────────────────────────────────────────────────┐
│                    DB Process                        │
│  ┌─────────────────────────────────────────────┐    │
│  │              Buffer Pool                     │    │
│  │  ┌───────┬───────┬───────┬───────┐         │    │
│  │  │Page 1 │Page 5 │Page 3 │Page 9 │ ...     │    │
│  │  └───────┴───────┴───────┴───────┘         │    │
│  │  - LRU/Clock 알고리즘으로 관리                │    │
│  │  - 페이지 교체 정책 직접 제어                  │    │
│  │  - Dirty page 추적                          │    │
│  └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘
           │
           │ read()/write() 시스템 콜
           ▼
┌─────────────────────────────────────────────────────┐
│                   OS Page Cache                      │
└─────────────────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────────────┐
│                      Disk                           │
└─────────────────────────────────────────────────────┘

문제: Double Buffering (데이터가 두 번 캐시됨)
해결: O_DIRECT 플래그로 OS 캐시 우회
```

**mmap 방식:**

```
┌─────────────────────────────────────────────────────┐
│                    DB Process                        │
│  ┌─────────────────────────────────────────────┐    │
│  │         Memory-mapped Region                 │    │
│  │    (가상 주소로 직접 접근)                     │    │
│  │    ptr[offset] = value;                      │    │
│  └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘
           │
           │ 직접 매핑 (Page Table)
           ▼
┌─────────────────────────────────────────────────────┐
│          OS Page Cache (= Buffer 역할)              │
└─────────────────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────────────────┘
│                      Disk                           │
└─────────────────────────────────────────────────────┘

장점: 단순함, 복사 없음
단점: OS가 페이지 교체 정책 결정 (DB가 제어 못함)
```

### 2.6 mmap의 DB에서의 문제점

**문제 1: 페이지 교체 제어 불가**

```
DB 관점에서 이상적인 교체:
- Query Plan을 알고 있음
- "이 페이지는 곧 다시 쓸 거야" 예측 가능
- 트랜잭션 우선순위에 따른 교체

OS 관점의 교체:
- LRU 기반 (최근 사용 안 된 것 교체)
- DB의 Query Plan 모름
- 중요한 인덱스 페이지를 내쫓을 수 있음
```

**문제 2: Page Fault의 예측 불가능성**

```
Buffer Pool:
┌────────────────────────────────────────┐
│ page = buffer_pool.get(page_id);       │
│ if (page == NULL) {                    │
│     // 명시적으로 I/O 수행             │
│     // 비동기 처리 가능                 │
│     // 타임아웃 설정 가능               │
│ }                                      │
└────────────────────────────────────────┘

mmap:
┌────────────────────────────────────────┐
│ value = mapped_region[offset];         │
│ // Page Fault 발생하면?                │
│ // - 현재 스레드 블록됨                 │
│ // - 언제 끝날지 모름                   │
│ // - 타임아웃 불가                      │
└────────────────────────────────────────┘
```

**문제 3: Error Handling**

```c
// read() 방식
ssize_t result = read(fd, buf, size);
if (result < 0) {
    // 에러 처리 가능
    perror("read failed");
}

// mmap 방식
char c = mapped_ptr[offset];
// I/O 에러 발생 시 → SIGBUS 시그널
// 일반적인 예외 처리로 잡기 어려움
```

**문제 4: Write Ordering (쓰기 순서)**

```
WAL (Write-Ahead Logging) 원칙:
"데이터 변경 전에 반드시 로그를 먼저 디스크에 기록"

mmap 문제:
┌─────────────────────────────────────────────────────┐
│ 1. 로그 페이지 수정 (메모리)                         │
│ 2. 데이터 페이지 수정 (메모리)                       │
│ 3. OS가 임의 순서로 flush 가능!                     │
│    → 데이터가 먼저 flush되면 WAL 위반!              │
└─────────────────────────────────────────────────────┘

해결: msync()로 명시적 flush + 순서 제어
      → 그러면 mmap의 단순함 장점 상실
```

```

└────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────┐
│                      mmap 사용                         │

┌────────────────────────────────────────────────────────┐

├────────────────────────────────────────────────────────┤
│ MongoDB (WiredTiger) - Buffer Pool + 일부 mmap        │
│ SingleStore(MemSQL)  - 컬럼스토어에 mmap               │
└────────────────────────────────────────────────────────┘
```

### 2.8 LMDB의 mmap 활용 (성공 사례)

```
LMDB 설계 원칙:
┌─────────────────────────────────────────────────────┐
│ 1. Single Writer, Multiple Readers                  │
│    - Writer가 Copy-on-Write로 새 페이지 생성         │
│    - Reader는 기존 페이지 계속 참조                   │
│                                                     │
│ 2. 읽기에는 Lock 없음                                │
│    - mmap된 영역을 그냥 읽으면 됨                     │
│    - Reader는 Page Fault만 감수                      │
│                                                     │
│ 3. Append-only B+Tree                               │
│    - 기존 페이지 수정 안 함                          │
│    - Write ordering 문제 회피                        │
└─────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────┐
│ Transaction T1 (읽기)     Transaction T2 (쓰기)      │
│        │                         │                   │
│        ▼                         ▼                   │
│   ┌─────────┐               ┌─────────┐             │
│   │ Root v1 │               │ Root v2 │ (새로 생성)  │
│   └────┬────┘               └────┬────┘             │
│        │                         │                   │
│   ┌────▼────┐               ┌────▼────┐             │
│   │ Page A  │               │ Page A' │ (COW 복사)   │
│   └─────────┘               └─────────┘             │
│        ▲                                            │
│        │                                            │
│   T1은 계속 v1 참조 (일관성 유지)                      │
└──────────────────────────────────────────────────────┘
```

## 3. 두 개념의 연결점 요약

```
┌─────────────────────────────────────────────────────────────┐
│                    Memory-mapped Index                       │
│                           │                                  │
│                    내부적으로 사용                            │
│                           ▼                                  │
│    ┌─────────────────────────────────────────────────┐      │
│    │              Lazy Loading 메커니즘               │      │
│    │                                                 │      │
│    │  • 전체 인덱스를 메모리에 올리지 않음              │      │
│    │  • 접근하는 페이지만 Page Fault로 로드            │      │
│    │  • OS가 Page Cache로 LRU 관리                   │      │
│    │  • 안 쓰는 페이지는 자동으로 swap out            │      │
│    └─────────────────────────────────────────────────┘      │
│                                                             │
│    Demand Paging + mmap = Memory-mapped Index의 핵심        │
└─────────────────────────────────────────────────────────────┘
```

**결론:** mmap 기반 인덱스는 OS의 Demand Paging을 활용한 Lazy Loading의 대표적 응용입니다. 단순하고 효율적이지만, DB의 세밀한 제어가 필요한 경우 Buffer Pool이 더 적합할 수 있습니다.

### 2. Memory-mapped Index 활용

블로그에서 직접 언급:

> "Since the index is memory-mapped from disk, the postings are lazily iterated and merged to get the final list."
> 

### 2.1 인덱스 구조와 mmap

```
Persistent Block의 Index 파일:
┌────────────────────────────────────────────────────────────┐
│                     index 파일 (disk)                       │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ Symbol Table │ Series │ Label Index │ Postings │ TOC │  │
│  └──────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────┘
                          │
                          │ mmap()
                          ▼
┌────────────────────────────────────────────────────────────┐
│                  Virtual Address Space                      │
│  index[0x7fff0000 ... 0x7fff9000]                          │
│                                                            │
│  접근 예: index[offset] → Page Fault → 디스크에서 로드       │
└────────────────────────────────────────────────────────────┘
```

### 2.2 Postings Offset Table의 메모리 최적화

블로그 내용:

> "Storing only some of the value helps in saving memory... every 32nd at the moment, including the first and the last label value"
> 

```
전체 저장 (비효율적):
┌──────────────────────────────────────────────────────┐
│ map[labelName][]postingOffset                        │
│                                                      │
│ "status" → [pos("200"), pos("201"), pos("202"), ... │
│             pos("500"), pos("501"), pos("502"), ...] │
│                                                      │
│ 수천 개의 값 = 메모리 낭비                             │
└──────────────────────────────────────────────────────┘

Prometheus 방식 (32개마다 샘플링):
┌──────────────────────────────────────────────────────┐
│ map[labelName][]postingOffset                        │
│                                                      │
│ "status" → [pos("200"),    ← 첫 번째                 │
│             pos("231"),    ← 32번째                  │
│             pos("263"),    ← 64번째                  │
│             ...                                      │
│             pos("502")]    ← 마지막                  │
│                                                      │
│ 나머지는? → mmap된 디스크에서 필요 시 탐색!            │
└──────────────────────────────────────────────────────┘
```

### 2.3 LabelValues(name) 쿼리에서의 활용

```
LabelValues("status") 쿼리 흐름:

1. 메모리에서 "status"의 첫/마지막 position 조회
   ┌─────────────────────────────┐
   │ first_pos = positions[0]   │  (메모리)
   │ last_pos = positions[-1]   │
   └─────────────────────────────┘

2. mmap된 인덱스에서 해당 구간 순회
   ┌─────────────────────────────────────────────┐
   │ for pos in range(first_pos, last_pos):     │
   │     value = index[pos]  ← Page Fault 가능!  │  (mmap)
   │     results.append(value)                   │
   └─────────────────────────────────────────────┘

   → 디스크 I/O는 실제 접근 시에만 발생
   → OS Page Cache가 자주 접근하는 부분 캐싱
```

---

### 3. Lazy Loading / Lazy Evaluation 활용

### 3.1 Postings 리스트의 Lazy Iteration

블로그 내용:

> "all the postings for all the matching entries are not got into the memory at the same time"
> 

```
job=~"app.*" 매처 처리:

Eager 방식 (비효율적):
┌────────────────────────────────────────────────────────┐
│ postings_app1 = load_all(index["job"]["app1"])  // 전체 로드
│ postings_app2 = load_all(index["job"]["app2"])  // 전체 로드
│ postings_app3 = load_all(index["job"]["app3"])  // 전체 로드
│ ...
│ result = merge(postings_app1, postings_app2, ...)
│
│ 문제: 수십만 개의 series ID가 한번에 메모리에!
└────────────────────────────────────────────────────────┘

Lazy 방식 (Prometheus):
┌────────────────────────────────────────────────────────┐
│ iterators = [                                          │
│     lazy_iter(index["job"]["app1"]),  // 아직 로드 안 함
│     lazy_iter(index["job"]["app2"]),  // 아직 로드 안 함
│     lazy_iter(index["job"]["app3"]),  // 아직 로드 안 함
│ ]                                                      │
│                                                        │
│ merged_iter = merge_iterators(iterators)               │
│                                                        │
│ // 실제 사용 시점에 하나씩 로드                           │
│ for series_id in merged_iter:                          │
│     process(series_id)  ← 이때 필요한 posting만 로드     │
└────────────────────────────────────────────────────────┘
```

### 3.2 Sample Iterator의 Lazy Loading

블로그 내용:

> "All the sample iterators for all series are not returned upfront... An iterator is returned which iterates over the series one by one giving its sample iterator. And the sample iterator also lazily loads the chunks when asked for."
> 

```
Select(job=~"app.*") 실행 흐름:

┌─────────────────────────────────────────────────────────────┐
│ Step 1: Select() 호출                                       │
│         → SeriesIterator 반환 (아직 데이터 로드 안 함)        │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ Step 2: SeriesIterator.Next() 호출                          │
│         → 첫 번째 series의 SampleIterator 반환               │
│         → 해당 series의 chunk 참조만 획득 (데이터는 아직)      │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ Step 3: SampleIterator.Next() 호출                          │
│         → 이때서야 chunks 디렉토리에서 실제 데이터 로드!       │
│         → mmap된 chunk 파일에서 Page Fault로 로드            │
└─────────────────────────────────────────────────────────────┘
```

**코드 레벨로 표현:**

```go
// Select는 즉시 반환 - 실제 데이터 로드 없음
seriesSet := querier.Select(matchers...)

// series 순회 시 하나씩 로드
for seriesSet.Next() {
    series := seriesSet.At()

    // sample 순회 시 chunk 로드
    iter := series.Iterator()
    for iter.Next() {
        ts, val := iter.At()  // ← 이 시점에 chunk 데이터 로드!
        // process sample
    }
}
```

### 3.3 다중 블록 쿼리의 Lazy N-way Merge

```
mint=1000, maxt=5000 쿼리가 3개 블록에 걸친 경우:

┌─────────┐  ┌─────────┐  ┌─────────┐
│ Block 1 │  │ Block 2 │  │ Block 3 │
│ 0-2000  │  │2000-4000│  │4000-6000│
└────┬────┘  └────┬────┘  └────┬────┘
     │            │            │
     ▼            ▼            ▼
┌─────────┐  ┌─────────┐  ┌─────────┐
│ Querier1│  │ Querier2│  │ Querier3│
└────┬────┘  └────┬────┘  └────┬────┘
     │            │            │
     └─────────┬──┴────────────┘
               │
               ▼
      ┌─────────────────┐
      │  Merge Querier  │
      │                 │
      │ Lazy N-way Merge│ ← 모든 블록 데이터를 한번에 로드하지 않음
      │                 │    각 블록에서 필요한 만큼만 순차적으로
      └─────────────────┘
```

### 5. 왜 이런 설계를 했는가?

### Prometheus TSDB의 특수한 상황

```
┌────────────────────────────────────────────────────────────┐
│ 도전 과제:                                                  │
│                                                            │
│ • 수백만 개의 시계열 (high cardinality)                      │
│ • 각 시계열당 수천 개의 샘플                                  │
│ • 실시간 쿼리 응답 필요                                      │
│ • 단일 서버에서 동작 (분산 시스템 아님)                        │
│ • 메모리 제한 환경에서도 동작해야 함                          │
└────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌────────────────────────────────────────────────────────────┐
│ 해결책:                                                     │
│                                                            │
│ Memory-mapped Index:                                       │
│ • 인덱스 전체를 메모리에 올릴 필요 없음                       │
│ • OS가 Page Cache로 hot data 자동 관리                     │
│ • 32비트 환경 고려 불필요 (시계열 DB는 보통 64비트)            │
│                                                            │
│ Lazy Loading:                                              │
│ • 쿼리가 실제 필요로 하는 데이터만 로드                       │
│ • Iterator 패턴으로 메모리 사용량 일정하게 유지               │
│ • 대량의 시계열도 streaming 처리 가능                        │
└────────────────────────────────────────────────────────────┘
```

### Head Block과의 대비

```
Head Block (최근 데이터, 메모리):
┌────────────────────────────────────────────────────────────┐
│ map[labelName]map[labelValue]postingsList                  │
│                                                            │
│ • 전체가 메모리에 상주                                       │
│ • mmap 불필요                                              │
│ • 빠른 쓰기/읽기 (실시간 ingestion)                          │
└────────────────────────────────────────────────────────────┘

Persistent Block (과거 데이터, 디스크):
┌────────────────────────────────────────────────────────────┐
│ Memory-mapped Index + Lazy Iteration                       │
│                                                            │
│ • 대부분 디스크에 존재                                       │
│ • 쿼리 시에만 필요한 부분 로드                               │
│ • 메모리 효율적 (수십 GB 데이터도 처리 가능)                  │
└────────────────────────────────────────────────────────────┘
```

---

### 6. 정리

| 개념 | Prometheus TSDB에서의 활용 |
| --- | --- |
| **Memory-mapped Index** | Index 파일을 mmap하여 Postings Offset Table, Series Table 등에 접근. 전체 인덱스를 메모리에 올리지 않고 OS Page Cache 활용 |
| **Lazy Evaluation** | Postings 리스트를 Iterator로 순회하며 merge. 전체를 메모리에 올리지 않고 streaming 처리 |
| **Lazy Loading** | Sample Iterator가 실제 `.Next()` 호출 시에만 chunk 데이터 로드. 다중 블록 쿼리 시 N-way merge도 lazy하게 수행 |

**핵심 인사이트:** Prometheus는 mmap과 lazy loading을 결합하여, 수백만 시계열과 수십 GB 데이터를 제한된 메모리에서도 효율적으로 쿼리할 수 있는 구조를 만들었습니다.
