# Checkpointing & Log Truncation & Compaction

데이터베이스 시스템이 데이터를 저장할 때, 무작정 쌓아두기만 하면 금세 저장 공간이 바닥나고 시스템 복구에 수 시간이 걸리게 됩니다. 이를 방지하기 위해 **Checkpointing, Log Truncation, Compaction** 세 가지 핵심 기법이 사용됩니다.

## **1. Checkpointing: 안전 점검 지점 만들기**

**Checkpointing**은 메모리에 있는 변경 사항을 디스크의 영구 저장소에 반영하고, "여기까지는 확실히 저장되었다"라고 기록하는 과정입니다.

- **필요성:** WAL(Write-Ahead Log)은 장애 복구를 위해 필수적이지만, 로그가 무한정 길어지면 재시작 시 로그를 읽는 시간(Replay time)이 너무 오래 걸립니다.
- **핵심 역할:** 장애 발생 시 복구 시작 지점을 결정합니다. 체크포인트 이전의 로그는 이미 디스크에 반영되었으므로 다시 읽을 필요가 없습니다.
- **다른 시스템에서의 예시:**
    - **MySQL:** 더티 페이지(Dirty Page)를 디스크로 플러시하고 LSN(Log Sequence Number)을 기록합니다.
    - **Redis:** RDB 스냅샷 생성 자체가 하나의 거대한 체크포인트 역할을 합니다.

## **2. Log Truncation: 불필요한 과거 삭제하기**

**Log Truncation**은 체크포인트가 완료되어 더 이상 복구에 필요 없어진 과거의 WAL 로그 파일을 물리적으로 삭제하거나 재사용하는 작업입니다.

- **동작 원리:** "체크포인트 완료 = 해당 구간 데이터의 안전 보장"이라는 공식이 성립될 때 실행됩니다.
- **이점:** 유한한 디스크 공간을 효율적으로 관리하고, 시스템 복구 시 탐색해야 할 로그의 범위를 최소화합니다.
- **판단 기준:** "이 로그 없이도 시스템 상태를 완벽히 복구할 수 있는가?"가 물리적 삭제의 핵심 기준이 됩니다.

## **3. Compaction: 읽기 성능을 위한 재구성**

**Compaction**은 흩어져 있거나 중복된 데이터 세그먼트들을 병합하고 정리하여 읽기 최적화된 구조로 만드는 과정입니다.

- **동작 원리:** 일단 빠른 쓰기를 위해 흩어져 저장된 데이터(LSM-Tree 방식 등)를 주기적으로 읽어 들여, 정렬하고 중복을 제거한 뒤 새로운 큰 블록으로 다시 저장합니다.
- **주요 작업:**
    - **중복 제거:** 동일한 키에 대해 가장 최신 값만 남깁니다.
    - **삭제 처리:** 삭제 마커(Tombstone)가 붙은 데이터를 실제로 물리적으로 삭제합니다.
    - **정렬 및 병합:** 여러 개의 작은 파일을 하나의 큰 파일로 합쳐 조회 시 발생하는 I/O 횟수를 줄입니다.

## **4. Prometheus TSDB에서의 실제 적용**

![image.png](https://github.com/user-attachments/assets/25a34caf-60a0-48ce-a0c1-ef75edd3e050)

프로메테우스는 시계열 데이터 특성에 맞춰 이 세 가지 개념을 정교하게 연결하여 사용합니다.

### **Head Block의 순환 구조**

프로메테우스의 인메모리 영역인 **Head Block**은 데이터가 쌓여 일정 범위(*chunkRange* * 1.5, 약 3시간)를 넘어서면 정리를 시작합니다.

- **Compaction (1단계):** 가장 오래된 2시간(*chunkRange*)치 데이터를 메모리에서 잘라내어 디스크상의 **Persistent Block**으로 변환합니다.
- **Checkpointing:** WAL에 쌓인 **Series 정보(라벨셋 ↔ series ID 매핑)** 를 별도의 checkpoint 파일로 compact하여 저장합니다. Redis의 RDB처럼 전체 메모리를 스냅샷하는 것과 달리, WAL의 메타데이터만 추려낸 **부분 스냅샷**입니다. 재시작 시 WAL 전체를 처음부터 replay하는 대신, checkpoint 파일을 먼저 로드한 뒤 그 이후의 WAL segments만 replay하면 되므로 복구 시간이 단축됩니다.
- **Log Truncation:** 새로운 블록과 체크포인트가 안전하게 저장되었으므로, 해당 구간의 **WAL 파일들을 삭제(Truncate)** 합니다.

### **Persistent Block의 병합**

- **Compaction (2단계):** 디스크에 저장된 블록들도 시간이 지나면 더 큰 단위로 병합됩니다. 2시간짜리 블록들이 모여 6시간, 18시간 단위로 점점 더 큰 블록으로 합쳐지며 쿼리 성능을 유지합니다. (블록 크기는 보존 기간 설정까지 계속 커질 수 있으며, 2주는 Prometheus 기본 보존 기간입니다.)

## **5. 요약: 데이터의 생애 주기**

| **단계** | **역할** | **목적** |
| --- | --- | --- |
| **WAL** | 기록 | 데이터 유실 방지 (쓰기 최적화) |
| **Checkpointing** | 확정 | 복구 범위 제한 및 상태 저장 |
| **Log Truncation** | 해제 | 저장 공간 확보 및 불필요 데이터 제거 |
| **Compaction** | 최적화 | 읽기 성능 향상 및 데이터 구조 정돈 |
